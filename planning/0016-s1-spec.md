## Why (context — the design note is gitignored/absent here, so it's inlined)

This is the prerequisite for the per-profile **circuit breaker** (0016 Slice 2): the breaker's
health signal will be "count of `kind:'correction'` events for a profile in a rolling window /
total events for that profile" — two `WHERE profile_id = ? AND ts > ?` count queries on
`audit_events`. **`audit_events` has no `profile_id` column today.**

The settled decision (don't re-litigate): **denormalize** `profile_id` onto `audit_events` —
do NOT plan to join `audit_events → sessions` to recover it. Reason: the volume-GC sweep
routinely `DELETE`s `sessions` rows while `audit_events` rows persist (no FK, no cascade), so a
join would silently lose profile attribution for exactly the long-tail historical data the
rolling-window health metric depends on. This mirrors how `team_id`/`user_id` are already
denormalized onto `audit_events`, and how `pull_requests` carries its own `profile_id`/`team_id`.

This slice is **capture only** — add the column, plumb it through, make it required so no call
site can forget it. It introduces **no** new behavior, query, or gate. The breaker (Slice 2)
consumes it.
